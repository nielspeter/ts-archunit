## Context

ts-archunit was built to enforce architecture rules that AI coding agents violate. The library already has the enforcement engine (body analysis, presets, rich violation messages) **and** a CLI that already surfaces it: `npx ts-archunit check --format json` emits structured violations, and `npx ts-archunit explain` dumps active rules. Two gaps remain between that surface and how agents actually work:

1. **Agents don't know constraints upfront.** The `explain` command outputs JSON and terminal markdown, but agents consume _imperative_ markdown in their system prompt / project instructions. There is no format an agent can read once and internalize as "rules I must not break," and no idempotent way to keep that block current.

2. **No preset targeting agent-specific mistakes.** Agents make predictable mistakes — inline logic, generic errors, empty stubs, copy-paste — that no single one-liner catches today. Assembling the relevant rules by hand requires reading the docs and knowing which function-variant conditions to reach for.

**Goal:** Two features that close the loop for agents, both riding the CLI/preset surface that already exists — upfront context via an agent-optimized explain format, and one-liner setup via an agent-aware preset — plus the docs that tie them into a self-correcting agent workflow.

### Why not an MCP server

The original draft led with an MCP server exposing `check_architecture` and `explain_rules` tools. Review and a data-shape audit showed it added no capability the CLI lacks: agents all have a terminal tool and can run `npx ts-archunit check --format json` / `explain` directly, and the only field the proposed MCP response added over the CLI JSON (`codeFrame`) is already computed on `ArchViolation` and one line away in `formatViolationsJson`. The single genuine differentiator was a warm ts-morph project cache (cold-start latency on large monorepos), which is a CLI-daemon concern, not an MCP one. The server is therefore deferred, not built — see "Deferred: MCP server (rejected for now)". Dropping it removes the `@modelcontextprotocol/sdk` peer dependency and keeps the core at two runtime deps (ts-morph, picomatch) per ADR-001, with no MCP-specific machinery (execution model, cache refresh, path validation, stdio safety) to maintain.

### The agent workflow this enables

1. **Setup (once):** author `arch.rules.ts` — often just `export default [...agentGuardrails(p, { ... })]` (the returning form) alongside any topology rules.
2. **Context (once per repo):** `npx ts-archunit explain arch.rules.ts --format agent >> CLAUDE.md` writes an imperative rule block the agent reads before writing a line. Re-running replaces the block in place.
3. **Loop (every edit):** the agent runs `npx ts-archunit check --format json` in its own terminal after writing code, reads the violations + suggestions, and self-corrects — before committing. No git push, no CI round-trip.

## Phase 1: Agent-Optimized Explain Format (0.5 day)

### What it does

Adds an `agent` output format to the existing `explain` command. Outputs imperative markdown optimized for system prompts / project-instruction files. Two parts: a **fixed check-in-loop preamble** (telling the agent to run `check --format json` after edits and fix what it reports — this is what actually closes the loop, not derived from the rules) and the **rules** rendered as imperative "Do NOT / MUST" bullets grouped by category.

```bash
npx ts-archunit explain arch.rules.ts --format agent >> CLAUDE.md
```

### Output format

```markdown
<!-- ts-archunit:start -->

## Architecture Rules (auto-generated by ts-archunit)

The following rules are enforced by CI. Violations will block your PR.

**Verify before you continue:** after writing or changing code, run
`npx ts-archunit check --format json`, read the `violations` array, and fix
each one using its `suggestion`. Do this in your edit loop — don't wait for CI.

### Layer Rules

- Do NOT import from `**/repositories/**` in files under `**/routes/**` — routes must go through services
- Do NOT import from `**/infrastructure/**` in files under `**/domain/**` — domain must be independent

### Code Patterns

- Do NOT throw `new Error()` in classes extending BaseRepository — use NotFoundError, ValidationError, or DomainError
- Do NOT call `parseInt` in classes extending BaseRepository — use this.extractCount() instead
- Do NOT call `eval()` anywhere in source code

### Hygiene

- Do NOT leave empty function bodies — every function must have at least one statement
- Do NOT leave stub comments in source code
- Every exported symbol must be referenced by at least one other file

### Required Patterns

- Functions in `**/services/**` MUST call a method matching /Repository/ — services must delegate to the data layer
<!-- ts-archunit:end -->
```

Sentinel markers (`<!-- ts-archunit:start -->` / `<!-- ts-archunit:end -->`) enable idempotent updates — running the command again replaces the block instead of appending. Consumers of the format (`explain --format agent`) write only what is between the markers; if the markers already exist in the target file, the writer must replace the enclosed span. (The command writes to stdout; the redirect/insert is the user's responsibility. Docs show the append pattern and note the idempotent-replace intent for tooling that pipes it into a managed block.)

### Imperative text generation

The primary path is a new optional `imperative` field on both `RuleMetadata` (input — what rule authors set) and `RuleDescription` (output — what `describeRule()` returns):

```typescript
// src/core/rule-metadata.ts — add imperative field
export interface RuleMetadata {
  id?: string
  because?: string
  suggestion?: string
  docs?: string
  imperative?: string // "Do NOT throw new Error() in repositories"
}

// src/core/rule-description.ts — add imperative field
export interface RuleDescription {
  rule: string
  id?: string
  because?: string
  suggestion?: string
  docs?: string
  imperative?: string
}
```

Built-in conditions provide their own `imperative` text via the condition's `description` field + the builder's predicate context. The heuristic string-matching is the **fallback**, not the primary path. This ensures custom rules degrade gracefully (raw description) while built-in rules produce polished imperative sentences.

To populate `imperative`, `RuleBuilder.describeRule()` is updated:

```typescript
describeRule(): RuleDescription {
  return {
    rule: this.buildRuleDescription(),
    id: this._metadata?.id,
    because: this._reason,
    suggestion: this._metadata?.suggestion,
    docs: this._metadata?.docs,
    imperative: this._metadata?.imperative ?? this.buildImperative(),
  }
}
```

Where `buildImperative()` applies the heuristic conversion as a fallback. Rule authors can also pass `imperative` directly via `.rule({ imperative: 'Do NOT ...' })`.

### What it does

A single preset targeting the specific mistakes AI agents make most often. One function call, no per-rule assembly. The other presets answer "where does code go?" (layer ordering, import direction); `agentGuardrails` answers "how is the code written?" — a bundle of body-analysis rules that a linter does not catch but ts-archunit does.

Uses **function variants** of all rules (not class-only versions) so standalone functions, arrow functions, and class methods are all covered — agents write all three.

```typescript
import { agentGuardrails } from '@nielspeter/ts-archunit/presets'

agentGuardrails(p, {
  src: 'src/**',
  noInlineLogic: ['parseInt', 'JSON.parse', 'eval'],
  noGenericErrors: true,
  noStubs: true,
  noEmptyBodies: true,
  noCopyPaste: true,
})
```

### Generated rules

All conditions below already exist in the tree — the preset composes them, it adds no new conditions.

| Rule ID                              | Entry point            | Condition (source)                                 | Default |
| ------------------------------------ | ---------------------- | -------------------------------------------------- | ------- |
| `preset/agent/no-inline-logic/<api>` | `functions(p)`         | native `.notContain(call(api))` (builder method)   | error   |
| `preset/agent/no-generic-errors`     | `functions(p)`         | `functionNoGenericErrors()` (`rules/errors.ts`)    | error   |
| `preset/agent/no-stubs`              | `functions(p)`         | `noStubComments()` (`rules/hygiene.ts`)            | error   |
| `preset/agent/no-empty-bodies`       | `functions(p)`         | `noEmptyBodies()` (`rules/hygiene.ts`)            | error   |
| `preset/agent/no-copy-paste`         | `smells.duplicateBodies` | `DuplicateBodiesBuilder` (a `TerminalBuilder`)   | warn    |

### Wiring — the returning form (plan 0060, Option 2)

`agentGuardrails` **returns an array of severity-carrying builders** (`RuleBuilderLike[]`), not `void` — so the agent's rules file does `export default [...agentGuardrails(p)]` and `check --format json` runs it through the unified pipeline, surfacing **every** violation as JSON, including the `no-copy-paste` **warn**. (A throwing/`console.warn` model would drop that warn from the agent's loop — the whole point.) Each rule is built, given its id + severity via `.rule({ id })` and the non-terminal `.asSeverity(effective)` (`effective = overrides[id] ?? defaultSeverity`; `'off'` skips it), and collected.

```typescript
export interface AgentGuardrailsOptions extends PresetBaseOptions {
  /** Glob for the source files the rules apply to. */
  src: string
  /** Banned call names — one rule generated per entry. */
  noInlineLogic?: string[]
  noGenericErrors?: boolean
  noStubs?: boolean
  noEmptyBodies?: boolean
  noCopyPaste?: boolean
}

export function agentGuardrails(p: ArchProject, options: AgentGuardrailsOptions): RuleBuilderLike[] {
  validateOverrides(options.overrides, collectRuleIds(options))
  const builders: RuleBuilderLike[] = []

  const push = (builder: FunctionRuleBuilder | DuplicateBodiesBuilder, id: string, def: RuleSeverity) => {
    const sev = options.overrides?.[id] ?? def
    if (sev !== 'off') builders.push(builder.rule({ id }).asSeverity(sev))
  }

  for (const api of options.noInlineLogic ?? [])
    push(functions(p).that().resideInFile(options.src).should().notContain(call(api)),
         `preset/agent/no-inline-logic/${api}`, 'error')
  if (options.noGenericErrors)
    push(functions(p).that().resideInFile(options.src).should().satisfy(functionNoGenericErrors()),
         'preset/agent/no-generic-errors', 'error')
  if (options.noStubs)
    push(functions(p).that().resideInFile(options.src).should().satisfy(noStubComments()),
         'preset/agent/no-stubs', 'error')
  if (options.noEmptyBodies)
    push(functions(p).that().resideInFile(options.src).should().satisfy(noEmptyBodies()),
         'preset/agent/no-empty-bodies', 'error')
  if (options.noCopyPaste)
    push(smells.duplicateBodies(p).withMinSimilarity(0.9),
         'preset/agent/no-copy-paste', 'warn')

  return builders
}
```

(`.rule({ id })` and `.asSeverity(level)` both return `this`; `.asSeverity` is plan 0060's non-terminal severity setter. A vitest user who wants throwing behavior spreads the array through a `.check()` helper.)

**Metadata is agent-facing (elided in the sketch).** Each `.rule()` call also carries `because` / `suggestion` / `imperative` — not just `id`. This is load-bearing for agent delivery: the `imperative` renders the rule as an actionable "Do NOT…" bullet in `explain --format agent`, and the `suggestion` is what the agent reads from `check --format json` to self-correct. A rule with only `{ id }` gives the agent "no-generic-errors violated" and no fix. (Same requirement as `recommended`, plan 0049.)

### Condition-wiring notes (grounded in the current API)

- **`.satisfy()` is the idiomatic bridge**, not a workaround. `functionNoGenericErrors()`, `noStubComments()`, and `noEmptyBodies()` are standalone conditions with no fluent wrapper on `FunctionRuleBuilder`; every test and doc applies them via `.satisfy(<condition>())` (e.g. `tests/integration/errors-functions.test.ts`, `tests/integration/hygiene-rules.test.ts`, `docs/standard-rules.md`). `.satisfy()` lives on the base `RuleBuilder` and accepts `Predicate<T> | Condition<T>`.
- **`noInlineLogic` uses the native `.notContain(call(api))`** builder method — no `.satisfy()` needed — because a fluent wrapper exists (`FunctionRuleBuilder.notContain`).
- **`noEmptyBodies` has two implementations; pick the condition.** There is both a `noEmptyBodies()` condition (`rules/hygiene.ts`, the documented standard rule) and a native `notHaveEmptyBody()` builder method. The preset uses the condition via `.satisfy(noEmptyBodies())` for consistency with the other three chains and the documented rule. Do NOT reach for `notHaveEmptyBody()` here — a subtle divergence (e.g. on arrow-function expression bodies `() => x`) would be invisible.
- **`noCopyPaste` default similarity.** `smells.duplicateBodies(p)` returns a `DuplicateBodiesBuilder` whose own default is `withMinSimilarity(0.85)`; the preset sets `0.9` explicitly (stricter, fewer false positives for a bundled default). Smell detectors default to `.warn()` by design, which is why this rule alone defaults to `warn`.

### `noInlineLogic` generates one rule per banned API

Each entry generates a separate builder with its own rule-ID suffix:

```
preset/agent/no-inline-logic/parseInt
preset/agent/no-inline-logic/JSON.parse
preset/agent/no-inline-logic/eval
```

This preserves specificity in violation messages — the agent sees which specific API was called, not a generic "banned API" message. `collectRuleIds()` must enumerate these per-API ids so `validateOverrides()` recognizes override keys like `preset/agent/no-inline-logic/parseInt`.

### Overrides

All rules accept per-rule severity via `overrides` (`PresetBaseOptions`):

```typescript
agentGuardrails(p, {
  src: 'src/**',
  noGenericErrors: true,
  overrides: {
    'preset/agent/no-generic-errors': 'warn', // downgrade to warning
    'preset/agent/no-empty-bodies': 'off', // disable
  },
})
```

### What's NOT in this preset

- `noConsole` / `noDefaultExports` — these overlap with ESLint rules most teams already have. The preset focuses on things only ts-archunit can catch (body analysis, delegation patterns, copy-paste detection).
- Class-only variants — the preset uses function variants so it catches violations in standalone functions, arrow functions, and class methods.

### Why a separate preset

`layeredArchitecture` enforces _where_ code goes. `agentGuardrails` enforces _how_ code is written. They compose in one rules file:

```typescript
// arch.rules.ts
export default [
  ...agentGuardrails(p, { src: 'src/**', noGenericErrors: true }),
  // + topology rules as builders (or a topology preset — layeredArchitecture
  //   still throws today; call it separately or await its returning form)
]
```

## Test strategy

### Agent format for explain (~12 tests)

- Generates markdown with imperative "Do NOT" / "MUST" sentences
- Uses `imperative` field from `RuleDescription` when available
- Falls back to heuristic conversion when `imperative` is not set
- Handles compound rule descriptions (multiple conditions)
- Preserves regex patterns in output (`/Repository/`)
- Incorporates `because` field as suffix
- Groups rules by category
- Empty rules produces "No rules found."
- Sentinel markers present for idempotent updates
- Custom `.rule({ imperative: '...' })` overrides heuristic
- Handles rules without metadata (raw description fallback)
- Markdown-special characters are escaped
- **Emits the fixed check-in-loop preamble** (instructs the agent to run `check --format json` after edits) — present even when there are zero rules

### Agent guardrails preset (~13 tests)

- Catches `eval` when `noInlineLogic` includes `'eval'`
- Each `noInlineLogic` entry generates a separate rule ID
- `noInlineLogic: []` (or omitted) is a no-op (no rules generated)
- Catches empty bodies when `noEmptyBodies: true` (via `noEmptyBodies()` condition)
- Catches stub comments when `noStubs: true`
- Catches generic `Error` when `noGenericErrors: true` (uses function variant — fires on arrow functions + methods, not just classes)
- Catches duplicate bodies when `noCopyPaste: true` (at 0.9 similarity)
- Passes for clean code with all options enabled
- **Returns `RuleBuilderLike[]`** — one builder per enabled rule, each carrying its `preset/agent/*` id and severity; `no-copy-paste` builder is `warn`, the rest `error`
- Override to `'off'` omits that builder from the returned array
- Override to `'warn'` sets the builder's `.asSeverity('warn')` (surfaces as warn, not a failure)
- Via the pipeline (plan 0060): a project tripping the rules → error rules exit 1, the copy-paste warn is reported in `--format json` at exit 0
- Optional rules skipped when not configured (e.g. `noGenericErrors` omitted)
- `src` glob scopes all generated rules
- Unknown override key warns via `validateOverrides` (typo guard), including per-API `no-inline-logic/*` ids

### CLI routing + agent loop payload (~3 tests)

- `await run(['explain', 'rules.ts', '--format', 'agent'])` produces agent-format output
- `explain`'s `--format agent` does not collide with the `check` command's `--format`
- `check --format json` includes `codeFrame` (plus `because`/`suggestion`) in each violation — the agent-loop payload

## Deferred: MCP server (rejected for now)

An MCP (Model Context Protocol) server exposing `check_architecture` + `explain_rules` tools was the original Phase 1. It is deferred, not built, because it duplicates the CLI without adding capability:

- **Data parity:** `check_architecture`'s response is a reshape of what `formatViolationsJson` already emits. The only extra field it proposed (`codeFrame`) is already computed on `ArchViolation` (`violation.ts`) and merely omitted by the serializer — a one-line change to add, not a missing feature. Two divergent JSON shapes (the MCP `summary` is a string; the CLI `summary` is `{ total, reason }`) would be a maintenance liability, not a benefit.
- **Access parity:** agents (Claude Code, Cursor, Windsurf, Cline) all have a terminal tool and can run `npx ts-archunit check --format json` / `explain` directly. MCP registration is friction the CLI does not require.
- **The one real differentiator was latency**, not the protocol: a long-running server holds a warm ts-morph `Program`, avoiding cold-start reload on every check. On small/medium projects this is negligible; on large monorepos it matters. But that is a **CLI-daemon** problem (`ts-archunit check --server` over a socket, no new dependency, works with any client), not an MCP-specific one.

**Cost avoided by deferring:** the `@modelcontextprotocol/sdk` peer dependency; `src/mcp/` (server + two tools + entry point); the `mcp` subcommand; and all MCP-only machinery (two-phase import/execute model, `refresh` cache invalidation, path validation, stdio JSON-RPC safety, cold-start handling) plus ~21 tests.

**Revisit when:** cold-start latency on a real target project (cmless-scale) is measured and shown to hurt the agent edit loop. At that point, evaluate a CLI daemon first, and only reach for MCP if a typed tool schema proves worth the dependency + per-client config. Track as a separate plan if pursued.

Also still out of scope:

- **Auto-fix / code transformation** — the agent applies fixes based on text suggestions. Structured fix templates (find/replace patterns) are a future enhancement.
- **Multi-project support beyond one tsconfig per invocation** — monorepos run the CLI per project (or via `workspace([...])`).
- **IDE plugins** — the CLI is the universal integration layer. No VS Code / JetBrains plugins.
- **Agent-specific rule tuning** — rules are the same for humans and agents. The preset just bundles common ones.
